# 「暴汗艺术家」前后端接口契约 v1.0

> 后端权威、前后端共享的接口约定。后端真实事实源是 `backend/api_server.py` 的 Pydantic 模型与
> `backend/agents/langgraph_agents.py` 的返回结构；本文档为其人类可读镜像，改接口时同步更新本文件。

- 后端地址：默认 `http://127.0.0.1:8000`
- 开发期前端走 `/api` 前缀，由 Vite 代理转发到后端（见 `frontend/vite.config.ts`），下表均以后端真实路径书写。
- 渲染分工：**后端只产出结构化数据，图表与语音播报由前端渲染**（前端自绘图 + 浏览器 TTS）。

---

## 1. 端点总览

| 方法 | 路径 | 用途 |
|---|---|---|
| POST | `/chat` | 任务型对话中枢：意图路由（报告/数据录入/偏题/安全熔断），返回结构化 JSON |
| POST | `/plan` | 一键生成健康决策报告（异步），返回 `task_id` |
| GET | `/status/{task_id}` | 轮询任务状态与报告结果 |
| GET | `/tasks` | 任务摘要列表 |
| GET | `/conversations` | 会话列表（取首条用户消息为标题，按更新倒序） |
| GET | `/conversations/{id}` | 某会话历史消息 |
| DELETE | `/conversations/{id}` | 清空某会话 |
| GET | `/health`、`/` | 健康检查 / 服务信息 |

---

## 2. POST `/chat`

任务型对话中枢。**只服务两类任务**：报告生成、数据录入；其余一律引导回这两类（`intent="other"`），命中安全红线则熔断（`intent="blocked"`）。

### 请求
```json
{
  "message": "帮我生成今天的健康体检报告",
  "conversation_id": "可选，为空则新建；响应回传本次会话ID",
  "image_base64": "可选，多模态图识别录入（饮食拍照/体脂秤拍照）",
  "mode": "可选，报告场景 control|experiment"
}
```

### 响应（ChatResponse）
| 字段 | 类型 | 说明 |
|---|---|---|
| `conversation_id` | string | 会话ID，前端需保存并在后续请求带上 |
| `intent` | string | `report` / `data_entry` / `other` / `blocked` |
| `data_type` | string\|null | 数据录入类型：`exercise` / `nutrition` / `body` |
| `reply` | string | 面向用户的话术（直接渲染到对话气泡） |
| `extracted` | object | 已提取/识别的字段 |
| `missing` | string[] | 仍缺失的关键字段（中文标签），非空表示需多轮补充 |
| `can_proceed` | bool | 数据是否齐全可继续 |
| `saved` | bool | 本轮是否已入库 |
| `task_id` | string\|null | `intent=report` 时返回，配合 `/status` 轮询 |

### 四种 intent 的前端处理

**① report** — 报告诉求 → 后端已异步起任务，前端拿 `task_id` 轮询 `/status`：
```json
{ "intent": "report", "task_id": "af36...", "can_proceed": true,
  "reply": "好的，正在为你生成…（task_id=af36…）" }
```

**② data_entry** — 数据录入（缺字段→多轮追问且**不入库**；齐全→已入库）：
```json
// 缺字段
{ "intent": "data_entry", "data_type": "exercise", "can_proceed": false, "saved": false,
  "missing": ["峰值心率(bpm)", "运动后60秒心率(bpm)"], "reply": "还需要你补充：…" }
// 齐全已入库
{ "intent": "data_entry", "data_type": "nutrition", "can_proceed": true, "saved": true,
  "extracted": {"diet_narrative": "早餐燕麦牛奶…"}, "reply": "✅ 已记录【饮食记录】：…" }
```
各类型必填字段：

| data_type | 必填字段 | 入库表 |
|---|---|---|
| `exercise` 运动负荷 | `duration_minutes`, `peak_hr`, `hr_60s`, `rpe` | `exercise_records` |
| `nutrition` 饮食记录 | `diet_narrative` | `nutrition_logs` |
| `body` 身体测量 | `weight_kg`, `body_fat_pct` | `users`（更新）|

> `date` 缺省取今天，无需用户提供；图识别字段由 `image_base64` 进来后并入 `extracted`（本期 mock）。

**③ other** — 偏题 → 礼貌引导，前端正常渲染 `reply` 即可。

**④ blocked** — 命中安全红线（如处方药/危险动作）→ **熔断就医分流话术**，前端建议用醒目红色样式（设计 §7 Guardrail Active）：
```json
{ "intent": "blocked", "can_proceed": false,
  "reply": "安全拦截警告 [Guardrail Active]：检测到提问涉及临床医学建议/药物指导…请寻求专业医生诊疗。" }
```

## 3. POST `/plan`（一键生成）

前端"一键生成报告"按钮直接调用，等价于 `/chat` 的 report 意图。

### 请求 / 响应
```json
// 请求
{ "user_id": 1, "mode": "control", "conversation_id": "可选，关联会话" }
// 响应
{ "task_id": "af36...", "status": "started", "message": "健康决策任务已启动，请用 task_id 轮询 /status" }
```
- `mode`：`control`（放任恶化，高疲劳演示）/ `experiment`（积极恢复）。
- `user_id`：暂无用户态，默认演示用户小明 `id=1`。

## 4. GET `/status/{task_id}`

### 响应（StatusResponse）
| 字段 | 类型 | 说明 |
|---|---|---|
| `task_id` | string | 任务ID |
| `status` | string | `started` / `processing` / `completed` / `failed` |
| `progress` | int | 0–100 |
| `message` | string | 阶段提示 |
| `result` | object\|null | 完成后为下方「报告结果对象」，否则 null |

### 报告结果对象（result）
```jsonc
{
  "success": true,
  "mode": "control",
  "fatigue_level": "high",                 // high | medium | low
  "physio_assessment": { "rs": 62, "fatigue": "high", "rhr_status": "elevated",
                         "insight": "…", "hrr": 12, "trimp": 810 },
  "training_plan": { "training_type": "recovery", "exercises": ["核心拉伸…"],
                     "reason": "…", "source": "wger_api" },
  "meal_plan": { "total_calories": 1850, "protein_target_g": 144, "diet_suggestion": "…",
                 "meals": [ {"meal":"早餐","time":"08:30","total_calories":271,"foods":[…]}, … ],
                 "sauce_compensation": 1.3, "source": "usda_api" },
  "sleep_advice": { "advice": "…", "focus": "…", "source": "rule_based" },
  "safety_result": { "status": "passed", "category": "safe", "level": "safe",
                     "violations": [], "warnings": [], "final_text": "" },
  "derived_metrics": { "rs": 62, "hrr": 12, "trimp": 810, "bmi": 23.5,
                       "tdee": 2600, "body_age": 35.0, "fatigue": "high" },
  "data_sources": { "watch": "db", "exercise": "mock", "diet": "mock", "body": "db" },
  "reasoning_chain": ["[数据接入] …", "[physio_evaluator] …", "[条件边] …", "…"],
  "agent_outputs": { "physio_evaluator": {...}, "exercise_coach": {...}, "…": {} },
  "final_report": { /* 见第 5 节 */ },
  "status": "completed"
}
```
- `reasoning_chain`：思维链日志，直接渲染为「智能体会诊室」时间线。
- `data_sources`：每项数据来源 `db`/`mock`/`default`，可在 UI 标注"真实/模拟"。
- `safety_result.status="blocked"` 时，`final_report.safety_blocked=true` 且 `vocal_narrative` 为就医熔断话术。

## 5. `final_report` 与 `chart_data`（前端渲染核心）

```jsonc
"final_report": {
  "visual_metrics": { "rs": 62, "body_fat": 24.0, "body_age": 35.0 },
  "vocal_narrative": "早上好，小明…本报告由AI生成，不能替代专业医疗诊断…",  // 前端 TTS 朗读
  "safety_blocked": false,                                              // 命中安全拦截时为 true
  "chart_data": { /* 下表四块 */ }
}
```

### `chart_data` 字段表

**① `rs_gauge`** — 准备度半圆仪表盘（设计 §5 状态仪）
| 字段 | 类型 | 说明 |
|---|---|---|
| `value` | number | RS 分值（指针位置）|
| `min` / `max` | number | 0 / 100 |
| `fatigue` | string | `high`/`medium`/`low`（决定指针所在色区）|
| `zones` | array | `[{label, range:[lo,hi], color}]` 三段色阶（红/黄/绿）|

**② `trend_dual`** — HRV & 身体年龄 7 日双曲线（设计 §5 推演趋势，答辩核心图）
```jsonc
"trend_dual": {
  "days": ["D1","D2","D3","D4","D5","D6","D7"],
  "hrv":      { "control": [45,42.8,…,32], "experiment": [45,…,45], "unit": "ms" },
  "body_age": { "control": [35,…,35.5],    "experiment": [35,…,34.6], "unit": "岁" },
  "legend": { "control": "放任恶化(灰虚线)", "experiment": "采纳建议(绿实线)" }
}
```
> 前端：`control` 画灰色虚线、`experiment` 画绿色实线，双指标可做上下两个子图。

**③ `radar_compare`** — 干预前后健康雷达（设计 §5 健康雷达）
```jsonc
"radar_compare": {
  "dimensions": ["睡眠","HRV","心率恢复","准备度","体脂(反向)"],
  "control":    [75,45,35,80.7,40],   // 干预前
  "experiment": [75,45,75,80.7,43.3], // 干预后
  "legend": { "control": "干预前", "experiment": "干预后" }
}
```
> 各维 0–100，越高越好（体脂为反向已换算）。两张雷达网重叠绘制。

**④ `cards`** — 今日训练 / 餐盘 / 睡眠 / 生理 / 健康建议三卡（结构化，直接渲染）
```jsonc
"cards": {
  "training": {                       // 运动建议面板
    "training_type": "recovery",
    "exercises": [ {"name":"核心拉伸 (Core Stretch)","muscles":["核心","脊柱"],
                    "sets":3,"duration_sec":30,"description":"…"},
                   {"name":"深蹲 (Squat)","sets":4,"reps":12,"muscles":["股四头肌"]} ],  // 组数×次数 / 组数×时长
    "recommended_duration": "40-60",   // 推荐运动时长(分钟)
    "calorie_target": "320-450",       // 单次消耗目标(千卡)
    "reason": "…" },
  "meal": {                            // 明日饮食建议面板（结构化三餐）
    "total_calories": 1850, "protein_target_g": 130,
    "meals": [ {"meal":"早餐","time":"08:30","total_calories":271,
                "foods":[{"name":"鸡蛋","grams":100,"calories":155,"protein":13.0}, …]}, … ],
    "day_total_calories": 1095, "day_total_protein": 117,
    "sauce_compensation": 1.3, "diet_suggestion": "…" },
  "sleep": { "advice":"…(可执行助眠建议)", "focus":"今晚优先改善的一点", "source":"rule_based" },
  "physio": {…physio_assessment},
  "health_advice": { "exercise":"…", "sleep":"…", "nutrition":"…" }   // UI"健康建议"三张文本卡
}
```
> 训练动作处方：力量类含 `sets`+`reps`，拉伸/恢复类含 `sets`+`duration_sec`，前端按字段渲染"4组×12次"或"3组×30秒"。

**⑤ `dashboard`** — 健康报告仪表盘各面板（来自数据库最近 14 天真实聚合）
```jsonc
"dashboard": {
  "kpi": { "health_score": 88, "score_delta_vs_last_week": 7, "status": "活力充沛",
           "exercise_rate": 100, "exercise_rate_delta": 4, "risk": "低风险" },
  "body_overview": { "heart_rate": 70, "weight_kg": 72.0, "bmi": 23.5, "bmr": 1678,
                     "body_fat_pct": 18.7, "muscle_mass_kg": 55.5, "update_date": "2026-06-10" },
  "sleep": { "score": 88, "total_hours": 8.3, "deep_sleep_percent": 22, "rem_sleep_percent": 22,
             "stages_timeline": [ {"hour":0,"deep":33,"light":20,"rem":7,"awake":5}, … ] },  // 浅睡/深睡柱状
  "nutrition": { "total_calories": 1794, "protein_g": 135, "carbs_g": 202, "fat_g": 54,
                 "ratios": {"protein":29,"carbs":44,"fat":26}, "balance_score": 88.3 },
  "exercise_today": { "steps": 9367, "steps_goal": 8000, "duration_minutes": 60,
                      "calories_burned": 512, "intensity": "中等" },
  "week_summary": { "workout_days": 5, "days_goal": 7, "total_calories_burned": 2478,
                    "avg_steps": 8602.6, "avg_sleep": 82.6 },
  "goals": { "steps_goal": 8000, "exercise_minutes_goal": 30, "exercise_duration_target": "40-60", … }
}
```
对应 UI 槽位：`kpi`→顶部四卡（综合评分/状态/达标率/风险，含较上周 delta）；`body_overview`→身体指标概览；
`sleep`→睡眠监测（`stages_timeline` 画分时柱状）；`nutrition`→饮食监测（`ratios` 画环形）；
`exercise_today`→运动监测；`week_summary`→本周运动/累计消耗；`goals`→各目标值。

> 说明：
> - `trend_dual` 用设计端点常量，**始终**呈现对照-实验分叉；`radar_compare` 数据库优先派生，跑
>   `backend/sql/seed_xiaoming.sql` 后各维自然分叉。
> - `dashboard` 来自 `backend/scripts/seed_week_history.py` 造的最近 14 天数据：**user=1 向好周**、
>   **user=2 恶化周**（演示可切 user_id 对比）。无历史数据时各字段回退 null。
> - 仍属 **agent 输出能力缺口**（seed 不解决，后续单独改）：结构化明日三餐(食物×克数×单品热量)、
>   训练方案的组数×次数、睡眠建议文本。

## 6. 会话历史接口

```jsonc
// GET /conversations -> 列表
[ { "conversation_id": "…", "title": "首条用户消息(≤30字)", "updated_at": 1749… } ]
// GET /conversations/{id} -> 历史
{ "conversation_id": "…", "messages": [ {"role":"user","content":"…"}, {"role":"assistant","content":"…"} ] }
// DELETE /conversations/{id} -> { "conversation_id": "…", "status": "cleared" }
```
- 后端持久化到 `ai_conversations`（按 `session_id`，归属演示用户 `id=1`）；同一会话行还存有该次报告的
  `training_plan/meal_plan/agent_decisions/safety_logs/speech_report`。

## 7. 安全语义（前端配合）

- `/chat` 返回 `intent="blocked"`：命中红线，**不会**进入数据录入或报告，前端用醒目告警样式渲染 `reply`。
- 报告内 `safety_result.status="blocked"` + `final_report.safety_blocked=true`：报告仍展示 `visual_metrics`，
  但 `vocal_narrative` 为就医分流话术；前端可在报告顶部加一条安全横幅。
- 红线包括：处方药/消炎药（双氯芬酸钠、布洛芬…）、激素、兴奋剂、病理诊断、极限断食、危险动作等。

## 8. 典型时序

```
报告：前端 POST /chat{"生成报告"} 或 POST /plan  ─►  拿 task_id
       └► 轮询 GET /status/{task_id} 直到 status=completed  ─►  渲染 result.final_report.chart_data + 朗读 vocal_narrative

录入：前端 POST /chat{"录入饮食…"}  ─►  missing 非空则继续追问 ; saved=true 则提示已记录
```
